# Identity Portfolio Course Outline

## Status

- Status: working content outline for the identity portfolio course
- Scope: the live identity route flow plus the supporting page system around it
- Primary research spine: `docs/_research/identity-system-core.md`
- Architecture basis: `docs/_specs/identity-portfolio-redesign/spec.md` and `docs/_specs/identity-portfolio-redesign/page-jobs.md`

## Purpose

Define the course as a page-by-page learning experience instead of a loose set of ideas.

This outline is meant to answer four questions clearly:

1. What are all the pages in the course?
2. What does each page need to do?
3. What order should students move through them?
4. What should students leave with after each step?

## Course Promise

Help a student turn a polished-but-generic portfolio into a clear public signal with visible proof, a stronger page shape, and a repeatable publishing loop.

## Audience

Primary audience:

- students
- early-career builders
- career starters with real work but weak portfolio framing

Secondary audience:

- instructors
- mentors
- maintainers using the site as a teaching system

## Learner Transformation

By the end of the course, a student should be able to:

1. identify the need their portfolio is speaking to
2. choose one dominant signal instead of mixing many identities
3. shape the first read of the page intentionally
4. place concrete proof close to the promise
5. build a homepage and supporting proof structure that feels coherent
6. publish the page into weak ties and use feedback to improve it

## Core Course Logic

The course follows the research pipeline directly:

Motivation -> Identity -> Perception -> Trust -> Action -> Deployment -> Opportunity

The student-facing memory version is:

1. Need
2. Signal
3. Style
4. Proof
5. Build
6. Publish
7. Repeat

## Canonical Vocabulary

This course should teach a small, repeated vocabulary instead of a huge glossary.

### Vocabulary set

1. Need: the human tension the page is answering
2. Signal: the dominant thing someone understands about you on the first read
3. First read: what lands before deep reading begins
4. Visual lane: the style logic shaping perception on the page
5. Proof block: the compact evidence cluster that earns trust
6. Trust move: a structural choice that lowers uncertainty honestly
7. Product shape: the form the work takes so another person can understand and use it
8. Page job: the one thing a page is responsible for doing
9. Action path: the sequence from attention to next step
10. Weak-tie signal: the version of your work people can remember, repeat, or share after a quick encounter
11. Deployment loop: publish, observe, refine, repeat
12. Shape drift: when a page accumulates parts that weaken the main signal

## Teaching Pattern

Each core page should teach through the same sequence:

1. page problem
2. story of the idea
3. concept name and definition
4. worked example
5. copyable template or worksheet
6. output the student leaves with
7. clear next move

## Overall Flow

### Entry flow

1. `/`
2. `/experiences/identity-portfolio/`
3. `/experiences/identity-portfolio/signal/`
4. `/experiences/identity-portfolio/style/`
5. `/experiences/identity-portfolio/proof/`
6. `/experiences/identity-portfolio/build/`
7. `/experiences/identity-portfolio/publish/`

### Branching flow

Students branch only when stuck.

- From signal: archetypes lab
- From style: examples, layouts, tokens
- From proof: examples, persuasion lab
- From build: recipes, example pages, diagnose
- From publish: examples, sources, system map

### Return rule

Every support page must send the student back to the core path.

Support pages help a decision. They do not become the main journey.

## Core Path Outline

## Page 1: Start Route

Route: `/experiences/identity-portfolio/`

Dominant job: start the student fast and make the core path obvious.

Student question: what is this course and where do I begin?

Story to teach here:

The story of the polished but invisible portfolio.

The student has good work. The page still says almost nothing. The problem is not a lack of effort. The problem is lack of signal.

Vocabulary introduced:

1. Need
2. Signal
3. Page job

Required sections:

1. Hero promise
2. One before-and-after or contrast example
3. Short explanation of the five-step path
4. Core path card list
5. Optional support routes only after the main path is visible
6. Primary CTA into signal

Must include:

1. one promise
2. one visible proof cue
3. one primary CTA
4. one sentence explaining why the current route split exists

Must not include:

1. full theory stack
2. research library
3. long vocabulary dump
4. optional labs before the main CTA

Student output:

Not a finished artifact yet.

The output is orientation:

1. I understand the five-step path.
2. I know I need to start with signal.

Exit condition:

Student clicks into signal.

## Page 2: Signal Route

Route: `/experiences/identity-portfolio/signal/`

Dominant job: help the student choose the need-signal pair that should govern the first read.

Student question: what should my portfolio say first?

Story to teach here:

The story of the student with many decent projects and no center of gravity.

They are not failing because they lack work. They are failing because the first read has no stable interpretive frame.

Vocabulary introduced:

1. Signal
2. Need
3. First read
4. Primary archetype
5. Mixed signal

Required sections:

1. Why first read matters
2. Audience choice
3. Need choice
4. Archetype as interpretive frame
5. One-sentence promise
6. Signal worksheet or template
7. Worked example of a good signal brief
8. Next move into style

Copyable student output:

The signal brief should be an explicit artifact.

Minimum fields:

1. Audience
2. Need
3. Primary archetype
4. Secondary archetype, optional
5. One-sentence promise
6. Proof angle
7. CTA direction

Must include:

1. audience
2. need
3. one dominant signal
4. a signal brief template
5. one worked example

Must not include:

1. full style library
2. publishing system
3. long source trail

Exit condition:

Student leaves with a finished signal brief and moves into style.

## Page 3: Style Route

Route: `/experiences/identity-portfolio/style/`

Dominant job: translate the chosen signal into first-read page decisions.

Student question: what should this feel like when someone lands on it?

Story to teach here:

The story of two pages with the same words but different first reads.

One feels clear and trustworthy. One feels noisy or confused. The difference is perception, not just taste.

Vocabulary introduced:

1. Visual lane
2. First read
3. Hierarchy
4. Contrast
5. Perception

Required sections:

1. Why perception changes meaning
2. Swiss versus brutalist contrast
3. How different lanes change trust and attention
4. Product-shape teaser: what kind of object the homepage is becoming
5. Translation from lane to page choices
6. Visual direction worksheet
7. Next move into proof

Copyable student output:

The style decision sheet should include:

1. Chosen visual lane
2. Headline tone
3. Hero image type
4. Proof placement choice
5. Density rule
6. CTA tone rule

Must include:

1. first-screen examples
2. one strong visual contrast
3. a translation from style to page choices
4. one output template the student can fill in

Must not include:

1. full psychology library
2. build workflow
3. reference archive

Exit condition:

Student leaves with a first-screen direction and moves into proof.

## Page 4: Proof Route

Route: `/experiences/identity-portfolio/proof/`

Dominant job: help the student choose the evidence that earns trust fastest.

Student question: what proof belongs near the promise?

Story to teach here:

The story of the smart page no one believed.

The copy sounded good. The page still felt empty because trust had to be inferred instead of shown.

Vocabulary introduced:

1. Proof block
2. Trust move
3. Receipt
4. Authority
5. Social proof

Required sections:

1. Why adjectives are weak proof
2. What a proof block is
3. Trust patterns used ethically
4. Artifact selection criteria
5. Filled proof-block example
6. Student proof-block template
7. Next move into build

Copyable student output:

The proof block should have at least:

1. Block title
2. Artifact name
3. What changed
4. Method or process note
5. Outcome or result
6. Caption tying proof back to the promise

Must include:

1. artifact selection
2. trust explanation
3. proof-block format
4. one complete example

Must not include:

1. archetype browsing
2. long research lecture
3. unrelated page examples with no proof lesson

Exit condition:

Student leaves with a drafted proof block and moves into build.

2. one conversation opener
3. one follow-up script

Must include:

1. publishing note
2. weak-tie explanation

2. archetype browsing

## Support Route Outline

Route: `/experiences/identity-portfolio/diagnose/`

Job: help the student name the current failure before they try to fix it.

Sections:

1. Symptom chooser
2. Common failure categories
3. Route recommendation map
4. Return CTA to the owning core route

Student uses this when:

The page feels wrong but the reason is unclear.

Job: show better page moves in context.

Sections:

1. Before-and-after stories
2. Annotated improvements
3. Comparison grid

5. Return CTA to the core route

Student uses this when:

They need receipts, not more theory.

## Assessment and Review Flow

The course should assess students through visible artifacts, not only reflection.

### Assessment moments

1. Signal review: is the first read coherent?
2. Style review: does the page feel like the chosen signal?
3. Proof review: does trust come from visible evidence?
4. Build review: does the page have a strong product shape?
5. Publish review: can the work travel into weak ties?

### Final package

The student should leave with:

1. signal brief
2. homepage draft
3. proof block
4. selected work section
5. CTA
6. post draft
7. follow-up script

## Flow Rules

1. Core path pages should be completed in order.
2. Support pages should solve one stuck state and return the student to the core path.
3. Every core page should end with one visible artifact output.
4. Every core page should introduce no more than two or three key vocabulary terms.
5. Every core page should teach at least one story of the idea, not only the definition.

## Future Content Expansion

Once the core path is fully scaffolded, the course can expand with:

1. one completed student example per major archetype
2. worksheet downloads or persistent workbook state
3. instructor notes for critique sessions
4. alternate pathways for different portfolio types
5. deeper story-led idea pages that explain where the concepts come from

## Bottom Line

The course should feel like a guided build, not a smart archive.

The student should move through a small number of purposeful pages, leave each page with a concrete artifact, and only branch into theory or examples when the main build path no longer gives them what they need.
