# Identity Portfolio Redesign Content Doctrine

## Purpose

Define the writing, pacing, and content-density rules for the redesign so the site sounds human and stays readable.

## Voice doctrine

### Core rule

Talk to the student directly.

The site should sound like a smart, grounded guide helping someone make a better page, not like a system explaining itself from outside.

### Required voice qualities

- direct
- natural
- specific
- concrete
- calm
- confident

### Disallowed voice qualities

- syllabus-like
- self-congratulatory
- implementation-meta
- abstract before concrete
- over-explained
- performatively smart

## Hard writing rules

1. Use second person when giving guidance.
2. State the page job in plain language within the first screen.
3. Prefer page problems and page fixes over framework summaries.
4. Use short, concrete labels before technical vocabulary.
5. Keep theory language only when it changes a design decision.

## Phrases and patterns to avoid

Do not write like this:

- “This route proves…”
- “Students should be able…”
- “The learner…”
- “This page demonstrates…”
- “The system now…”
- “The important shift is…”

Write like this instead:

- “Use this page when…”
- “Start here if…”
- “Do this first.”
- “If your page feels vague, try this.”
- “Keep this. Cut that.”

## Density rules

### First-screen rule

The first screen may contain:

1. one promise
2. one proof or example cue
3. one primary CTA
4. one optional secondary route

It may not contain a full framework stack.

### Section rule

Each major section should introduce only one real idea.

If a section needs to teach more than one of these at once, split it:

- the problem
- the principle
- the example
- the fix
- the next step

### Theory rule

Theory pages must use this order:

1. page problem
2. why it happens
3. what to change
4. concrete example
5. next move

No theory page should read like a reference article before it reads like help.

## Example rules

Every core or lab page should include at least one of these:

1. before-and-after page comparison
2. weak version versus stronger version
3. vague line versus improved line
4. artifact with annotation
5. problem card with matching fix card

### Minimum example requirement by page type

- Start page: one clear before-and-after or one sharp example
- Signal page: one promise rewrite example
- Style page: one visible contrast like Swiss versus brutalist
- Proof page: one artifact block example
- Build page: one finished output example
- Publish page: one public post or follow-up example
- Lab page: one page problem and one fix per major concept

## One-job page checklist

Before a page is approved, confirm:

1. I can say what this page is for in one sentence.
2. I can say what the student should do after reading it.
3. I can point to the concrete example that earns the page.
4. I can name the content that does not belong here.

If any answer is unclear, the page is doing too much.

## Forbidden states

1. Repeating the pitch across multiple top sections.
2. Making the student scroll through optional theory before the main build step.
3. Using callouts as storage bins for dense framework content.
4. Using faces, diagrams, or novelty visuals without a page job.
5. Hiding the main CTA inside a cluster of equal-priority blocks.
6. Explaining the architecture when the student needs a page move.

## Student-read test

Every redesigned route should pass this honest read test:

1. Can a student skim the first screen and know why they are here?
2. Can they find the next action without reading the whole page?
3. If they read top to bottom, does the page stay on one job?
4. Does the page sound like help instead of commentary?
