## Homepage Sections

### Section 1: What I help with

I am strongest when a team has good material but weak structure.

I help with:

- turning large research or strategy docs into page flows people can use
- making a homepage or experience route communicate one clear signal instead of five mixed ones
- moving proof closer to the claim so a page earns trust faster
- repairing release-aware QA when copy, route scope, and tests drift apart
- making workflow and release logic easier to inspect, explain, and maintain

### Section 2: Selected work

#### Project 1: Identity Portfolio System Route Split

Turned a dense identity portfolio experience into a five-step student path.

What changed:

- split one overloaded route into signal, style, proof, build, and publish
- moved examples, labs, system map, and sources off the main path
- aligned route copy to a single research model instead of competing documents

Why it matters:

The page stopped asking students to read the whole theory stack before they could start making their portfolio.

Proof:

- route-level copy now maps to one dominant job per page
- start page now drives the student directly into the signal step
- the research hierarchy was tightened into core doc, handout, doctrine, evidence base, and blueprint

#### Project 2: Release-Aware Browser Baseline Repair

Repaired browser regressions caused by release-aware parked routes and stale expectations.

What changed:

- added real page-level headings to parked top-level routes for accessibility
- updated Playwright checks to branch between live and parked route behavior
- aligned recipe, primitives, accessibility, and status expectations with the current selected release

Why it matters:

The site can now intentionally park routes without silently breaking accessibility or QA.

Proof:

- full browser suite returned to green
- 25 tests passed
- 5 tests skipped by design

#### Project 3: Workflow and Release Systems for Multi-Experience Publishing

Helped shape the repo's file-backed workflow, release automation, and observable-state direction.

What changed:

- formalized experience, unit, visual, review, and release concepts as checked-in artifacts
- supported release-aware validation instead of implicit latest-content behavior
- contributed to the repo's release and observability planning chain so product state can be inspected, not guessed

Why it matters:

The system is moving from one-off site building toward a durable publishing workflow.

Proof:

- explicit release manifests, review records, and QA docs exist in the repo
- workflow and observability specs now describe stable artifact lineage instead of ad hoc process memory
