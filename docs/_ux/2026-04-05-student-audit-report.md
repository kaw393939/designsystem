# Student Portfolio Audit Report

## Audit setup

I audited the site as if I were a student trying to use the live identity experience to build a real portfolio draft from scratch.

I started from the home page, followed the core path in order, and forced each route to produce a concrete artifact instead of just notes.

The resulting portfolio draft is in `docs/_ux/2026-04-05-student-portfolio-example.md`.

## Main finding

The route split works. The student path is much clearer than a single giant theory page.

But the site currently teaches decisions more clearly than it teaches outputs.

I was able to finish a portfolio example because I already know how to translate abstract prompts into working artifacts. A less experienced student would likely stall between the signal route and the build route because the site does not yet provide enough scaffolding for the intermediate deliverables.

## What worked

### 1. The homepage gets me into the right path fast

The home page and identity start page do a good job of telling me that the real work starts in the identity experience, not in the support docs.

That is an improvement over a site that asks for too much reading before the student can act.

### 2. The core route order makes sense

The sequence is easy to understand:

1. Signal
2. Style
3. Proof
4. Build
5. Publish

That order feels natural and matches how a real portfolio is actually made.

### 3. The voice is more useful than most portfolio-teaching sites

The writing is direct and practical. It sounds like help, not like a lecture trying to impress me.

### 4. The build page is the strongest page in the flow

It is the first place where the site clearly names outputs, checks, and a usable sequence from brief to live page.

### 5. The publish route is strategically important

Treating posts, meetups, and follow-ups as part of the portfolio system is one of the best ideas in the whole experience.

## Where I got stuck or had to invent my own scaffolding

### 1. The signal route did not give me a copyable worksheet

The signal route clearly tells me to choose audience, need, and signal.

What it does not give me is a simple copy-paste form like:

- Audience:
- Need:
- Primary archetype:
- One-sentence promise:
- Proof angle:
- CTA:

I had to invent that structure myself.

### 2. The style route is inspiring but still too interpretive

The Swiss versus brutalist contrast is strong, but I still had to make a lot of practical translation decisions on my own.

I needed a stronger bridge from visual lane to actual homepage choices such as:

- headline length
- image type
- proof placement
- section count
- CTA style

### 3. The proof route explains the principle but not the artifact format

The route convinced me that proof should sit near the promise, but it did not hand me a reusable proof-block pattern.

I had to invent my own structure for:

- block title
- artifact name
- method
- outcome
- caption

### 4. The build route arrives too late with the clearest scaffolding

The build route is excellent, but it is the first route that really feels like a production worksheet.

By that point I had already made several important decisions without a template.

### 5. The publish route needs ready-to-edit scripts

The route gives the right categories: publishing note, meetup opener, follow-up script.

But again, I had to write those from scratch.

The site should provide one editable baseline example for each.

## Route-by-route student experience

### Home page

Good.

It correctly behaves like a bridge and gets me into the identity path quickly.

### Identity start page

Good.

It explains the five-step path clearly and keeps optional depth secondary.

### Signal

Important and conceptually clear, but under-scaffolded.

It tells me what to decide, not exactly what to write down.

### Style

Strong teaching page.

Still needs a more concrete translation into page modules and copy shape.

### Proof

Strong principle page.

Needs a reusable block template and at least one filled example.

### Build

Best page in the flow.

It names outputs, sequence, and quality gates clearly.

### Publish

Strategically strong, but still a little too high-level.

It needs editable scripts and maybe one complete before-and-after deployment example.

### Examples

Useful, but they feel slightly too far away from the moments when I needed them.

I would have used them earlier if the core pages pointed to a specific example right when I got stuck.

## Concrete artifacts I was able to produce

By following the route sequence, I was able to create:

1. a signal brief
2. a chosen archetype and visual lane
3. a homepage hero and supporting copy
4. a proof block
5. three project summaries
6. a CTA
7. a short public post
8. a meetup opener
9. a follow-up script

That means the system is strong enough to produce real output.

The problem is not that it fails. The problem is that it still makes the student do too much invisible synthesis.

## Highest-priority improvements

### 1. Add a persistent student workbook

This is the biggest improvement.

Each core route should fill part of one shared workbook with fields the student can copy, edit, and carry forward.

Minimum sections:

1. audience
2. need
3. archetype
4. promise
5. visual lane
6. proof block
7. CTA
8. publish note

### 2. Add a copyable signal brief directly on the signal page

Do not make the student infer the template.

Give them the exact structure.

### 3. Add a page-pattern translation block on the style page

For each visual lane, show:

1. what the hero should look like
2. what kind of proof should appear first
3. how dense the page should feel
4. what kind of CTA fits that lane

### 4. Add a filled proof-block example on the proof page

One strong example would reduce a lot of uncertainty.

### 5. Add a copyable homepage skeleton on the build page

Something like:

1. eyebrow
2. headline
3. dek
4. proof strip
5. selected work
6. CTA

### 6. Add ready-to-edit scripts on the publish page

Give the student:

1. one post template
2. one meetup opener template
3. one follow-up template

### 7. Pull examples closer to the core path

Each route should point to one exact example at the moment of likely confusion.

Not just “open examples,” but “if you picked Sage, open this example” or “if your proof feels vague, look at this exact block.”

## Lower-priority improvements

### 1. Add one fully completed student portfolio example inside the core path

Not just in support docs.

The student should see what “done enough” looks like before they reach the final route.

### 2. Add estimated effort per route

Students need help knowing whether a page is a 10-minute choice page or a 90-minute working session.

### 3. Add route-end checkpoints that produce a visible artifact

Example:

- signal route output: signal brief
- style route output: first-screen direction
- proof route output: proof block draft
- build route output: homepage stack
- publish route output: deployment kit

## Bottom line

The redesign succeeded at the biggest structural problem: the site no longer behaves like one giant theory page.

I could use it to build a real portfolio example.

But I could do that only by supplying my own worksheets, artifact formats, and output structure between routes.

The next improvement should not be more theory.

The next improvement should be stronger scaffolding for the artifacts the student is supposed to leave with.
