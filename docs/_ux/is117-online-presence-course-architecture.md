## Purpose

Turn IS117 into a coherent studio course where students do not merely learn web mechanics or receive vague portfolio advice.

They build a real public-facing online presence that functions as:

1. a professional web presence
2. a self-promotion tool
3. a proof surface
4. the first visible artifact in the larger BSEAI degree story

## Working claim

IS117 should be taught as an online presence studio, not as a disconnected introduction to web syntax and not as a loose “personal branding” exercise.

The student is building one product across the term:

their own online web presence and self-promotion system.

That product should be credible enough to use outside class.

## The problem this document solves

The current identity route system is much better than a giant theory page, but the audit showed a clear weakness:

The site teaches decisions more clearly than it teaches outputs.

That is not good enough for IS117.

IS117 needs:

1. a stronger assignment ladder
2. a persistent workbook or output trail
3. clearer intermediate deliverables
4. a tighter bridge from archetype and design thinking into an actual built site

## Course promise

By the end of IS117, each student should leave with a working version of a professional online presence that helps the right people understand them quickly, trust what they can do, and know what next step to take.

## The student is actually making

The course should name the product clearly.

The student is making:

1. a personal website or portfolio presence
2. a first-screen signal about who they help and how
3. a proof-backed self-promotion tool
4. a public-facing career artifact they can keep using after the course ends

This should not be framed as fake influencer branding.

It should be framed as a public professional presence built from real signal, real proof, and real follow-through.

## Instructional philosophy for IS117

### 1. One semester, one product

The course should revolve around one product that gets clearer, stronger, and more public over time.

### 2. The portfolio system is the course backbone

The portfolio system is not an extra unit.

It is the main spine because it gives the course a clear student-facing product.

### 3. Archetype and gallery modules exist to improve the assignment

They are not side libraries.

They should help students make better decisions about signal, style, trust, and proof.

### 4. The room is a studio; the site is the instructional operating system

Face-to-face time should be used for:

1. demonstration
2. critique
3. workshop time
4. feedback and revision

The site should own:

1. weekly clarity
2. templates
3. assignment instructions
4. examples
5. support materials
6. continuity across meetings

### 5. Publish is required, not optional

The product is not finished when the page is coded.

The student must also learn how to circulate and use it.

## IS117 course architecture in one line

Students move from current web presence confusion to a clear public identity, then build and publish a proof-backed website they can use to promote themselves.

## Signature assignment architecture

IS117 should center one signature assignment with three scaffolded phases.

### Signature assignment

`Build Your Online Presence System`

The complete assignment asks the student to create a public-facing site that includes:

1. a clear audience-facing promise
2. a coherent visual lane
3. visible proof of capability
4. a real call to action
5. a basic deployment kit for promoting the site

### Phase A: Presence strategy pack

This is the first major deliverable.

It should happen early enough that students do not drift into building before they know what they are building.

Required outputs:

1. current presence audit
2. audience and need statement
3. primary archetype choice
4. one-sentence promise
5. proof inventory
6. CTA direction
7. visual lane choice
8. homepage section sketch

### Phase B: Web presence build v1

This is the middle build phase.

Required outputs:

1. homepage skeleton
2. hero section
3. proof block
4. selected work or capability section
5. CTA block
6. working site draft

### Phase C: Publish and promote loop

This is the deployment phase.

Required outputs:

1. short public post
2. meetup or conversation opener
3. follow-up script
4. revised site after early feedback

## The first deliverable that fixes the current weak spot

The first deliverable should directly address the current under-scaffolding problem.

### Assignment 1

`Online Presence Strategy Pack`

### Why this must exist

Students currently have to invent too much invisible synthesis between the signal route and the build route.

This assignment should make the intermediate work visible.

### Required fields

1. current website or web presence audit
2. target audience
3. audience need
4. primary archetype
5. optional secondary archetype
6. one-sentence promise
7. visual lane choice
8. proof inventory
9. first CTA direction
10. homepage section sketch
11. shadow or failure mode to avoid

### Inputs that should feed it

1. portfolio system start route
2. signal worksheet
3. archetype gallery compare mode
4. visual-design literacy translation mode
5. proof inventory prompt

### Success condition

A student should be able to hand this document to an instructor or peer and get meaningful critique before they write code.

## Strategy-pack crosswalk

The strategy pack should not remain an isolated planning document.

It should drive later build choices directly.

| Strategy-pack field | Later build surface | What the instructor should be able to critique |
| --- | --- | --- |
| audience and need | homepage hero and site framing | whether the site is speaking to a real person with a real need |
| primary archetype | headline tone, proof style, CTA tone | whether the first read feels coherent rather than mixed |
| one-sentence promise | hero copy and intro language | whether the promise is legible and specific |
| visual lane | hierarchy, image direction, section pacing | whether the page reads in the intended visual register |
| proof inventory | proof block and selected work section | whether the site earns trust with real evidence |
| CTA direction | CTA block and follow-up flow | whether the next move is credible and usable |
| homepage section sketch | homepage skeleton and build plan | whether the student can translate strategy into structure |

## Hero and the Outlaw translation for IS117

The Hero and the Outlaw material should make the archetype week more structured and more serious.

The archetype work should use four cluster rooms rather than twelve disconnected cards.

1. `Yearning for Paradise`: Innocent, Explorer, Sage
2. `Leaving a Thumbprint on the World`: Hero, Outlaw, Magician
3. `No Man or Woman Is an Island`: Regular Guy or Gal, Lover, Jester
4. `Providing Structure to the World`: Caregiver, Creator, Ruler

The week should then end with a positioning decision, not with browsing.

Each student should leave with:

1. one primary archetype
2. tone cues
3. proof-style cues
4. CTA-style cues
5. one shadow or trap to avoid

## The role of the archetype module in IS117

The archetype module is important, but it should be used correctly.

### What it should do

1. help students choose one coherent first-read signal
2. help them decide tone, proof style, and CTA style
3. reduce mixed-identity confusion early in the process

### What it should not do

1. become a personality test
2. delay the build process with endless browsing
3. replace audience or proof decisions

### IS117 rule

Students should leave the archetype work with one decision artifact:

`My online presence should read as this kind of person, for this kind of audience, because this is the most credible and useful frame for my work.`

The project note adds a second rule.

The archetype layer should help students decide headline tone, proof style, and CTA style, not just choose a label.

## The role of the portfolio system in IS117

For IS117, the portfolio system should supply:

## Weekly rhythm for IS117

This course should be taught in four instructional phases.

The early-week theory-to-action rule should stay visible.

By the end of week 2, students should already have a current-presence audit and a working direction for the semester product.

By the end of week 4, they should already hold a usable signal brief and archetype decision that can guide the build.

| Phase | Week band | Main question | Core modules | Main outputs |
| --- | --- | --- | --- | --- |
| 1. start and define | weeks 1-4 | what am I building and who is it for | orientation, why-now, portfolio start, signal, archetype decision | start-here completion, presence audit, signal brief, archetype decision sheet |

| 3. build and revise | weeks 7-11 | how do I turn this into a real site | build-and-product-shape plus coding implementation | homepage skeleton, site v1, critique revisions |
| 4. publish and promote | weeks 12-15 | how do I use this site in public | publish-and-weak-ties plus showcase wrapper | publish kit, revised site, final reflection |

## Multimedia expectations for IS117

The project docs make multimedia a teaching method rather than extra polish.

IS117 does not need every asset immediately, but it should use the gallery media intentionally.

### Minimum use in the first wrapper

1. annotated comparison boards in archetype and visual-design weeks
2. one core diagram in trust and proof week

### Student-to-student

## Deployment scenario model for IS117

The project deployment scenarios add a more believable publish layer.

Instead of telling students to “network” in the abstract, the course should teach five room types.

1. builder room
2. workshop room
3. networking room
4. governance room
5. career room

During the publish phase, students should choose at least one room type and prepare:

1. the one-sentence signal for that room
2. the proof artifact they would show there
3. the 24-hour follow-up note they would send after the interaction

This keeps the publish module grounded in real deployment patterns rather than vague advice.

## Accessibility and support requirements for IS117

The site should model the same quality it asks students to build.

IS117 should require:

1. accessible headings and readable structure in course materials
2. alt text and captions when students use media
3. legible contrast and hierarchy in the example work and the student sites
4. descriptive link text
5. clear tool guidance for publishing and submission
6. support links for technology and course help

IS117 should also enforce one public-proof rule.

Students should not submit a polished page shell without at least one concrete proof surface that a stranger can understand quickly.

## Outside-reviewer test and public-proof rubric

Each final site should survive a one-minute read by an outside reviewer.

Use this test during critique and final review.

| Question | What the reviewer should be able to infer quickly |
| --- | --- |
| who is this person for | target audience or opportunity space |
| what do they help with | audience-facing promise or role direction |
| why should I believe them | proof block, examples, or concrete capability signals |
| what should I do next | CTA and contact or follow-up path |

The public-proof rule for IS117 is therefore:

Every final project should show a clear promise, one coherent signal, at least one meaningful proof surface, and one credible next move.

## Measures of success for IS117

The course is working if:

1. students can say who their site is for and what it promises by week 3
2. students have a usable strategy pack before they begin major coding work
3. students can build a believable proof block rather than only a generic about page
4. the final site can actually be used outside class as a self-promotion tool
5. students leave with not just a website but a way to talk about and circulate it

## Bottom line

IS117 should be the first applied studio in the BSEAI system.

Its main deliverable is not “a webpage” in the thin sense.

Its main deliverable is a public-facing online presence and self-promotion tool built through the portfolio system, clarified by the archetype and gallery modules, strengthened by proof and design literacy, and completed by real deployment into the world.
